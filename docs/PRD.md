# PRD: Interview Evidence

## 1. 제품 정의

Interview Evidence는 RTZR STT로 전사한 2인 면접에서 사용자가 입력한 평가 기준과 의미적으로 관련된 지원자 답변을 찾아, 원문과 원음으로 재확인하게 하는 로컬 Streamlit 도구다.

제품이 답해야 하는 문장은 다음과 같다.

> 이 평가 기준을 검토할 때 다시 들어볼 만한 구간은 어디인가?

지원자를 자동 평가하거나 점수·순위·합격 가능성을 만드는 제품이 아니다.

## 2. 문제와 목표

면접 검토자는 긴 녹음을 다시 들으며 특정 평가 기준의 근거를 찾아야 한다. 정확한 표현을 기억하지 못하면 문자열 검색만으로는 관련 답변을 놓치기 쉽고, 요약만으로 판단하면 원래 문맥과 말투를 확인하기 어렵다.

MVP의 목표는 다음과 같다.

- RTZR 일반 STT API의 화자 분리와 타임스탬프를 정확히 사용한다.
- 자연어 평가 기준으로 지원자 답변을 의미 검색한다.
- 관련 답변 최대 3개를 원문과 원음에 연결한다.
- 다른 개발자가 README만으로 로컬에서 재현할 수 있는 범위를 유지한다.
- 평가 판단은 사용자에게 남기고 검색 결과를 평가 점수처럼 표현하지 않는다.

## 3. 대상 사용자와 사용 조건

### 주요 사용자

- 한 명의 지원자와 한 명의 면접관으로 진행한 녹음을 다시 검토하는 면접 담당자
- 인터뷰 답변에서 특정 역량이나 경험의 근거를 빠르게 찾으려는 사용자
- RTZR 파일 STT와 로컬 의미 검색의 결합 예제를 살펴보려는 개발자

### 지원 조건

- 한 번에 음성 파일 1개
- RTZR 결과의 고유 화자 2명
- 두 화자 중 사용자가 지원자 1명을 지정
- 자연어 질의는 한 번에 1개지만 같은 전사 결과에서 반복 검색 가능
- 로컬 실행을 기본으로 하며 배포는 MVP 범위가 아님

## 4. 핵심 사용자 흐름

UX는 다음 세 depth와 결과 상세 layer로 나눈다.

1. **음성 준비**: 사용자가 면접 음성 파일 한 개를 선택하고 실제 RTZR 전사를 시작한다.
2. 앱이 `transcribe_id`를 유지하며 완료될 때까지 상태를 폴링한다.
3. **화자 확인**: 완료된 전사에서 두 화자의 대표 발화와 전체 타임라인을 확인한다.
4. 사용자가 지원자 화자 한 명을 선택하고 나머지 화자는 면접관으로 확정한다.
5. 앱이 지원자의 연속 발화를 답변 구간으로 구성하고 검색 가능한 구간을 임베딩한다.
6. **근거 찾기**: 사용자가 평가 기준 또는 찾고 싶은 내용을 자연어로 입력한다.
7. 앱이 관련 답변을 최대 3개까지 순위로 표시한다.
8. **결과 상세 layer**: 사용자가 결과 하나를 선택해 전사 문맥과 답변 시작부터 종료 3초 후까지의 원음을 확인한다.
9. 사용자는 질의를 바꿔 다시 검색하거나 현재 질의와 결과를 JSON으로 다운로드한다.

각 depth는 한 가지 결정만 다룬다. 이전 depth로 돌아가도 전사 결과는 유지하며, 새 음성이나 지원자 화자 변경으로 무효화되는 이후 상태만 초기화한다.

## 5. 기능 요구사항

### FR-1. 실행 전 준비

- 앱은 `RTZR_CLIENT_ID`, `RTZR_CLIENT_SECRET`, `HF_TOKEN`을 `.env`에서 읽는다.
- EmbeddingGemma는 앱 실행 중 내려받지 않는다.
- 모델 캐시가 없으면 `python scripts/prepare_model.py` 실행 방법을 안내하고 의미 검색을 시작하지 않는다.
- 자격 증명이 없거나 잘못되면 실제 값을 화면에 출력하지 않고 설정 방법을 안내한다.

### FR-2. 음성 업로드와 RTZR 전사

- 한 번에 음성 파일 한 개만 받는다.
- 기본 전사 설정은 다음과 같다.

```json
{
  "model_name": "sommers",
  "domain": "GENERAL",
  "use_diarization": true,
  "diarization": { "spk_count": 2 },
  "use_word_timestamp": false,
  "use_itn": true,
  "use_disfluency_filter": true,
  "use_paragraph_splitter": true,
  "paragraph_splitter": { "max": 130 }
}
```

- 전사 요청 후 `transcribe_id`를 보존하고 기본 5초 간격으로 조회한다.
- `transcribing`, `completed`, `failed` 상태를 구분해 표시한다.
- 완료 응답의 `utterances`, `spk`, `start_at`, `duration`, `msg`를 검증한다.

### FR-3. 전사 타임라인과 화자 역할

- 발화를 시작 시각 순서로 화자, 타임스탬프, 텍스트와 함께 표시한다.
- 고유 화자가 정확히 두 명이면 사용자가 한 명을 지원자로 선택한다.
- 화자 선택 화면에는 역할 판단을 돕는 대표 발화 2~3개를 화자별로 표시한다.
- 대표 발화는 시간순 앞부분에서 지나치게 짧거나 빈 발화를 제외해 고르며 역할을 자동 추론하지 않는다.
- 나머지 화자는 자동으로 면접관으로 취급한다.
- 화자가 한 명이거나 세 명 이상이면 전사 타임라인은 보여주되 화자 지정과 의미 검색을 비활성화한다.

### FR-4. 답변 구간 구성

- 선택한 지원자의 연속된 발화를 하나의 답변 구간으로 묶는다.
- 다른 화자의 발화가 나타나면 현재 구간을 종료한다.
- 면접관의 짧은 맞장구도 MVP에서는 구간 경계로 처리한다.
- 각 구간은 결합 텍스트, 시작·종료 밀리초, 원본 발화 인덱스를 보존한다.
- 종료 시각은 마지막 발화의 `start_at + duration`으로 계산한다.
- 공백과 문장부호를 제거한 텍스트가 10자 미만이면 타임라인에는 유지하되 검색 대상에서 제외한다.
- 짧은 구간은 앞뒤 답변에 자동 병합하지 않는다.

### FR-5. 의미 검색

- 모델은 `google/embeddinggemma-300m`을 사용한다.
- 평가 기준은 `Retrieval-query`, 답변은 `Retrieval-document` 프롬프트로 인코딩한다.
- 모든 임베딩은 정규화한 뒤 코사인 유사도로 정렬한다.
- 답변 임베딩은 지원자 선택 후 한 번 계산하고 같은 세션에서 재사용한다.
- 질의 변경 시 질의 임베딩과 현재 결과만 갱신하고 검색 기록을 누적하지 않는다.
- 최소 유사도 임계값 없이 상위 3개를 표시한다.
- 검색 가능한 구간이 3개 미만이면 존재하는 결과만 표시한다.
- 기본 화면은 순위를 표시하고 정확한 유사도는 상세 영역에서만 제공한다.
- 다음 주의 문구를 결과와 함께 표시한다.

> 입력한 기준과 의미가 가까운 순서로 표시했습니다. 필요한 판단은 원문과 원음을 확인한 뒤 내려주세요.

### FR-6. 근거 원음 재생

- 결과마다 답변 시작·종료 타임스탬프를 표시한다.
- 사용자가 결과를 선택하면 근거 찾기 depth 안의 상세 layer를 열고 공용 오디오 플레이어의 재생 구간을 갱신한다.
- 상세 layer에는 선택한 답변, 직전 면접관 발화를 포함한 전사 문맥, 재생 범위와 접힌 검색 정보를 표시한다.
- 정확한 코사인 유사도와 원본 발화 인덱스는 `검색 정보`에서만 제공한다.
- 시작은 `max(0, floor(start_at_ms / 1000))`이다.
- 종료는 `min(audio_duration, ceil(end_at_ms / 1000) + 3)`이다.
- 자동 재생은 요구하지 않는다. 사용자가 직접 재생을 시작한다.

### FR-7. JSON 다운로드

- 현재 질의와 현재 Top 3만 UTF-8 JSON으로 다운로드한다.
- 서버 파일을 먼저 만들지 않고 메모리에서 `st.download_button`으로 전달한다.
- 결과 형식은 다음 필드로 제한한다.

```text
audio_filename
query
created_at
candidate_speaker
results[]
  rank
  similarity
  text
  start_at_ms
  end_at_ms
  source_utterance_indexes
```

- 음원, 전체 전사문, RTZR 원본 응답, 인증 정보는 내보내지 않는다.

### FR-8. 화면 depth와 카피

- 화면은 `음성 준비`, `화자 확인`, `근거 찾기` 순서로 전환한다.
- 세 depth를 한 화면에 동시에 노출하지 않는다.
- 상단에 현재 위치를 텍스트와 얇은 스카이블루 선으로 표시한다.
- 완료 퍼센트, 게임형 stepper와 별도 multipage 라우팅은 사용하지 않는다.
- `AI 분석`, `인사이트 생성`, `최적의 답변`, `지원자 적합도`처럼 자동 판단을 암시하는 문구를 사용하지 않는다.
- 주요 행동 문구는 `전사 시작`, `지원자로 선택`, `관련 답변 찾기`, `이 구간 듣기`, `전사 문맥 보기`, `현재 검색 결과 내려받기`를 사용한다.
- 결과 안내는 `입력한 기준과 의미가 가까운 순서로 표시했습니다. 필요한 판단은 원문과 원음을 확인한 뒤 내려주세요.`를 사용한다.

## 7. 비기능 요구사항

### 재현성

- 새 가상환경에서 README의 설치·준비·실행 순서가 동작해야 한다.
- 테스트 fixture는 실제 응답과 같은 파서와 후처리 경로를 사용해야 한다.
- 네트워크 없는 단위 테스트는 RTZR 및 Hugging Face 자격 증명을 요구하지 않아야 한다.

### 개인정보와 보안

- 자격 증명과 토큰은 `.env`에만 저장하고 Git에서 제외한다.
- 실제 면접 데이터는 저장소, fixture, 스크린샷과 예제 출력에 포함하지 않는다.
- 공개 샘플과 fixture는 합성 데이터만 사용하고 식별 메타데이터를 제거한다.
- 사용자 데이터와 검색 기록을 서버에 자동 저장하지 않는다.

### 성능과 자원

- Apple Silicon에서는 MPS를 우선 사용하고 실패 시 CPU를 사용한다.
- 초기 임베딩 배치 크기는 8을 기준으로 한다.
- 모델 인스턴스는 Streamlit 리소스 캐시로 재사용한다.
- 같은 화자 선택과 전사 결과에서는 답변 임베딩을 다시 계산하지 않는다.

### 화면과 문장

- 흰색과 회색을 기본으로 하고 스카이블루를 주요 동작, 현재 depth와 선택 상태에 사용한다.
- 주요 스카이블루는 `#0284C7`, hover는 `#0369A1`, 선택 배경은 `#E0F2FE`를 기준으로 한다.
- 검색 순위와 유사도에 성공·경고·오류 색을 연결하지 않는다.
- 화면 문장은 기능을 과장하지 않고 사용자의 다음 행동을 평범한 한국어로 설명한다.
- 색만으로 depth, 화자, 상태와 선택을 구분하지 않는다.

## 8. MVP 완료 조건

- 합성 2인 면접 WAV를 실제 RTZR API로 전사할 수 있다.
- 완료된 전사 결과에서 두 화자를 표시하고 지원자를 선택할 수 있다.
- 정해진 규칙대로 답변 구간을 구성하고 10자 미만 구간을 검색에서 제외한다.
- 대표 질의의 기대 답변이 Top 3 안에 포함된다.
- 선택한 답변의 시작 시점부터 종료 3초 후까지 원음을 재생할 수 있다.
- 현재 질의와 결과만 포함한 JSON을 내려받을 수 있다.
- 세 depth 사이를 이동할 때 전사 결과와 필요한 상태가 보존되고, 지원자 화자 또는 새 음성 변경 시 종속 상태가 초기화된다.
- 흰색·회색·스카이블루 색상과 합의된 고정 문구가 모든 depth에서 일관된다.
- 실패 상태가 사용자에게 복구 가능한 형태로 안내된다.
- README 절차가 새 환경에서 재현되고 비밀정보가 저장소와 Git 이력에 없다.

## 9. MVP 제외 사항

- 여러 지원자 비교, 자동 평가·점수·순위·합격 판단
- 질문 자동 탐지, 질문과 답변 자동 연결
- 여러 평가 기준의 일괄 분석과 검색 기록 누적
- 문자열 전용 검색, 북마크, 근거 채택·제외 편집
- 챕터 또는 질문·답변 경계 편집기
- 자동 키워드 추출과 단어별 타임스탬프
- Markdown, TXT, SRT 내보내기
- 저장된 RTZR 응답을 사용하는 사용자용 샘플 모드
- 계정, 데이터베이스, 실제 면접 데이터 영속화
- 실시간 마이크 전사, 별도 프론트엔드, 웹 배포

## 10. 선택 기능

핵심 MVP가 완료된 뒤에만 키워드 부스팅 입력을 고려한다.

- 기본값은 빈 목록이다.
- 전사 전에 회사명·기술 용어를 한 줄에 하나씩 입력한다.
- 빈 줄과 중복을 제거한다.
- `sommers` 모델에서는 한글 발음 표기를 사용한다.
- 항목당 최대 20자, 전체 최대 500개를 검증한다.
- 키워드 변경은 재전사가 필요함을 안내한다.
