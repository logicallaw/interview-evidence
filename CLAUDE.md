# 프로젝트: Interview Evidence

RTZR 일반 STT와 로컬 EmbeddingGemma를 이용해, 면접 평가 기준과 관련된 지원자 답변 구간을 찾아 원음으로 재확인하게 하는 근거 기반 인터뷰 리뷰어다. 자동 평가·점수화 도구가 아니다.

## 문서 우선순위

구현 전에 다음 문서를 읽는다.

1. `docs/PRD.md`: 제품 범위와 완료 조건
2. `docs/ARCHITECTURE.md`: 모듈 경계, 데이터 흐름, 오류 처리
3. `docs/ADR.md`: 확정된 기술 결정과 트레이드오프
4. `docs/UI_GUIDE.md`: 화면 상태와 표현 규칙
5. `interview-evidence-plan.md`: 결정 배경과 공식 문서 링크를 포함한 원본 기획

문서가 충돌하면 위 순서를 따른다. 구현 중 새 결정을 내렸다면 코드에만 남기지 말고 관련 문서도 함께 갱신한다. 아직 구현되지 않은 목표를 완료된 기능처럼 문서화하지 않는다.

## 기술 스택

- Python 3
- Streamlit
- RTZR 일반 STT API: 인증, 파일 전사 요청, 결과 폴링
- Sentence Transformers + `google/embeddinggemma-300m`
- PyTorch: Apple MPS 우선, CPU 대체
- pytest
- 환경 변수 로드: `.env`

## CRITICAL 제품 규칙

- 이 앱은 답변의 관련성 후보를 찾을 뿐 지원자를 평가하거나 합격 가능성을 계산하지 않는다.
- MVP는 음성 파일 1개, 정확히 두 화자, 지원자 한 명, 질의 한 개씩, 관련 답변 최대 3개만 지원한다.
- 사용자 앱은 항상 실제 RTZR API를 사용한다. 저장된 RTZR 응답은 테스트 fixture로만 사용하고 샘플 실행 모드로 노출하지 않는다.
- 의미 검색 결과에는 최소 유사도 임계값을 두지 않는다. 검색 가능한 구간이 3개 미만이면 존재하는 결과만 반환한다.
- 유사도는 정렬용 값이다. 기본 화면은 순위를 강조하고, 정확한 값은 상세 정보에만 표시한다.
- 선택한 근거의 오디오는 답변 시작 3초 전부터 종료 3초 후까지만 재생한다. 음원 경계를 넘지 않도록 보정한다.

## CRITICAL UI 및 카피 규칙

- UX는 `음성 준비 → 화자 확인 → 근거 찾기`의 세 depth로 나눈다. 세 depth의 기능을 한 화면에 동시에 노출하지 않는다.
- 결과의 전사 문맥과 공용 오디오 플레이어는 별도 페이지가 아니라 `근거 찾기` depth 안의 상세 layer로 연다.
- 별도 Streamlit multipage 라우팅을 만들지 않고 `st.session_state`로 현재 depth를 전환한다.
- 이전 depth로 돌아갈 때 전사 결과는 유지한다. 지원자 화자가 바뀌면 세그먼트·임베딩·검색·상세 상태를, 새 음성이 선택되면 전사 이후 상태 전체를 초기화한다.
- 화자 확인에는 화자별로 지나치게 짧지 않은 대표 발화 2~3개를 표시한다. 대표 발화로 역할을 자동 추론하지 않는다.
- 흰색과 회색을 기본으로 하고 스카이블루를 주요 버튼, 현재 depth와 선택 상태에만 집중해 사용한다.
- 기본 토큰은 main `#0284C7`, hover `#0369A1`, selection background `#E0F2FE`, border `#E2E8F0`, text `#0F172A`다.
- `AI 분석`, `인사이트 생성`, `최적의 답변`, `높은 관련성`, `지원자 적합도`, `AI 점수`라는 사용자 문구를 사용하지 않는다.
- 주요 문구는 `전사 시작`, `지원자로 선택`, `관련 답변 찾기`, `찾아볼 만한 답변`, `이 구간 듣기`, `전사 문맥 보기`, `현재 검색 결과 내려받기`를 사용한다.
- 검색 결과에는 `입력한 기준과 의미가 가까운 순서로 표시했습니다. 필요한 판단은 원문과 원음을 확인한 뒤 내려주세요.`를 표시한다.
- 색상, 게이지, 메달, 별과 점수로 결과의 품질을 표현하지 않는다.

## CRITICAL RTZR 규칙

- 요청 설정은 `model_name=sommers`, `domain=GENERAL`, `use_diarization=true`, `spk_count=2`, `use_word_timestamp=false`, `use_itn=true`, `use_disfluency_filter=true`, `use_paragraph_splitter=true`, `paragraph_splitter.max=130`을 기본으로 한다.
- 완료 응답을 사용하기 전에 `utterances`, `spk`, `start_at`, `duration`, `msg` 구조를 검증한다.
- 고유한 `spk`가 정확히 두 개가 아니면 타임라인은 표시할 수 있지만 화자 선택과 의미 검색은 비활성화한다.
- 전사 요청의 네트워크 타임아웃은 자동 재업로드하지 않는다. 서버 접수 여부가 불명확해 중복 작업이 생길 수 있다.
- 전사 요청의 `429 A0002`만 2초, 4초, 8초 간격으로 최대 세 번 재시도한다.
- 폴링은 기본 5초 간격이다. 폴링 중 일시적 네트워크 오류, 429, 500은 백오프 후 동일한 `transcribe_id` 조회를 계속한다.
- 60분 폴링 제한 도달은 작업 실패로 단정하지 않는다. `transcribe_id`를 보존하고 조회 재개 경로를 제공한다.
- 키워드 부스팅은 핵심 MVP가 아니다. 추가할 경우 업로드 전에 선택 입력으로 받고 `sommers` 규칙에 맞는 한글 발음, 항목당 20자 이하, 최대 500개를 검증한다.

## CRITICAL 검색 단위 규칙

- `utterances`를 `start_at` 오름차순으로 정렬한다.
- 선택한 지원자의 연속 발화를 하나의 답변 구간으로 묶는다. 다른 화자가 나오면 구간을 종료한다.
- 면접관의 짧은 맞장구도 우선 경계로 취급한다. 실제 fixture 검증 없이 맞장구 병합 휴리스틱을 추가하지 않는다.
- 구간에는 원본 발화 인덱스, 시작 시각, 종료 시각과 결합 텍스트를 보존한다.
- 종료 시각은 마지막 발화의 `start_at + duration`이다.
- 공백과 문장부호를 제거한 텍스트가 10자 미만인 구간은 타임라인에는 남기되 검색에서 제외한다. 앞뒤 구간에 자동 병합하지 않는다.
- 평가 기준은 `Retrieval-query`, 답변 구간은 `Retrieval-document` 프롬프트로 임베딩하고 모두 정규화한다.
- 답변 임베딩은 지원자 선택 후 한 번 계산해 같은 전사 세션에서 재사용한다. 질의 변경 시 질의 임베딩과 Top 3만 갱신한다.

## 아키텍처 규칙

- `app.py`는 현재 depth의 화면 렌더링, 입력과 `st.session_state` 조정만 담당하는 얇은 표현 계층으로 유지한다.
- RTZR HTTP 호출은 `src/rtzr_client.py`, 답변 구간 구성은 `src/segments.py`, 임베딩과 검색은 `src/semantic_search.py`, 내보내기는 `src/exporters.py`에 둔다.
- 외부 API 응답 딕셔너리를 UI에서 직접 순회하지 않는다. 검증·정규화된 도메인 데이터로 변환한 뒤 사용한다.
- 순수 로직은 Streamlit에 의존하지 않게 작성한다. 세그먼트 구성, 시간 경계, 정렬, JSON payload는 단위 테스트 가능해야 한다.
- 모델은 `scripts/prepare_model.py`로 미리 내려받고 짧은 임베딩까지 검증한다. 앱에서는 `local_files_only=True`와 `st.cache_resource`를 사용한다.
- MPS 로드 또는 추론이 실패하면 CPU로 한 번만 대체하고 실행 장치 변경을 사용자에게 알린다.
- 사용자 검색 기록을 누적하거나 데이터베이스를 추가하지 않는다.

## 개인정보와 비밀정보

- `RTZR_CLIENT_ID`, `RTZR_CLIENT_SECRET`, `HF_TOKEN`은 로컬 `.env`에서만 읽는다.
- 실제 자격 증명을 코드, 로그, 예외 메시지, fixture, 스크린샷, README와 Git 이력에 남기지 않는다.
- `RTZR_CLIENT_SECRET`은 보고서나 이메일에도 넣지 않는다.
- 브라우저 다운로드는 메모리의 UTF-8 JSON 바이트에서 직접 생성한다. 서버에 사용자 결과를 자동 저장하지 않는다.
- 내보내기에는 `audio_filename`, `query`, `created_at`, `candidate_speaker`와 Top 3 결과만 포함한다. 음원, 전체 전사문, RTZR 원본 응답과 인증 정보는 포함하지 않는다.
- `outputs/`는 로컬 디버깅용으로만 허용하며 Git에서 제외한다.
- 공개 fixture와 샘플은 합성 데이터만 사용하고 식별 정보와 불필요한 메타데이터를 제거한다.

## 개발 프로세스

- 한 번에 하나의 모듈 또는 사용자 흐름만 변경한다. 요청받지 않은 기능과 추상화를 추가하지 않는다.
- 각 변경 전에 검증 가능한 완료 조건을 정하고, 순수 로직과 오류 분기는 pytest로 검증한다.
- 실제 RTZR 호출은 fixture 기반 단위 테스트와 분리한다. fixture와 실제 응답은 동일한 파서와 후처리 경로를 사용해야 한다.
- API·모델 접근이 필요한 테스트는 명시적으로 구분해 기본 단위 테스트가 자격 증명이나 네트워크를 요구하지 않게 한다.
- 기존 코드 스타일을 따르고 관련 없는 리팩터링이나 정리를 하지 않는다.
- 기능을 추가해도 `docs/PRD.md`의 MVP 제외 사항을 묵시적으로 확장하지 않는다.
- 커밋 메시지는 Conventional Commits 형식(`feat:`, `fix:`, `docs:`, `test:`, `refactor:`)을 사용한다.

## 검증 명령

구현 파일이 준비된 단계부터 다음 명령을 사용한다.

```bash
python -m pytest
python scripts/prepare_model.py
streamlit run app.py
```

- `python -m pytest`: 자격 증명과 네트워크 없이 단위 테스트가 통과해야 한다.
- `python scripts/prepare_model.py`: Hugging Face 접근, 모델 로드, 짧은 임베딩 생성을 검증한다.
- `streamlit run app.py`: 실제 RTZR 자격 증명으로 전체 사용자 흐름을 수동 검증한다.

제출 전에는 대표 질의의 기대 구간이 Top 3에 포함되는지, 오디오 경계가 올바른지, JSON에 허용 필드만 들어가는지, 저장소와 Git 이력에 비밀정보가 없는지 확인한다.
