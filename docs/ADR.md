# Interview Evidence Architecture Decision Records

## 상태와 원칙

아래 결정은 제출용 MVP의 확정 기준이다. 변경할 경우 해당 ADR의 상태와 영향받는 PRD·아키텍처·UI 문서를 함께 갱신한다.

핵심 가치의 우선순위는 다음과 같다.

1. RTZR API의 정확한 사용
2. 다른 개발자가 재현할 수 있는 작은 범위
3. 사람이 원문과 원음으로 검증할 수 있는 근거
4. 개인정보와 비밀정보의 최소 보유
5. 기능 수보다 명확한 문서와 테스트 가능한 경계

---

## ADR-001: 파일 기반 RTZR 일반 STT 사용

**상태**: 승인
**결정**: MVP는 실시간 스트리밍이 아니라 `POST /v1/transcribe`와 `GET /v1/transcribe/{TRANSCRIBE_ID}`를 사용하는 파일 전사 흐름으로 구현한다.
**이유**: 한 개의 면접 녹음을 다시 검토하는 사용 사례에 맞고, 업로드·폴링·완료 응답이라는 작은 흐름으로 재현 가능하다.
**트레이드오프**: 실시간 자막과 마이크 입력은 지원하지 않는다. 긴 파일은 전사 완료까지 기다려야 한다.
**결과**: 앱은 `transcribe_id`를 세션에 보존하고 기본 5초 간격으로 상태를 조회한다.

## ADR-002: Streamlit을 얇은 UI 계층으로 사용

**상태**: 승인
**결정**: Python 기반 Streamlit 앱으로 만들되, 별도 multipage 라우팅 없이 세션 상태로 화면 depth를 전환한다. 외부 API, 세그먼트, 검색과 내보내기 로직은 `src/` 모듈로 분리한다.
**이유**: 제출 기한 안에 실행 가능한 UI를 만들면서 핵심 로직을 독립적으로 테스트할 수 있다.
**트레이드오프**: 세밀한 프론트엔드 상호작용과 복잡한 라우팅은 포기한다.
**결과**: `app.py`는 현재 depth의 렌더링과 이벤트 조정만 담당하며 RTZR 원본 JSON 해석, 임베딩 계산과 export payload 조립을 하지 않는다.

## ADR-003: MVP를 정확히 두 화자의 인터뷰로 제한

**상태**: 승인
**결정**: RTZR 요청에 `spk_count=2`를 사용하고 완료 결과의 고유 `spk`가 두 개일 때만 한 명을 지원자로 선택하게 한다. 나머지는 면접관으로 처리한다.
**이유**: 준비된 샘플과 핵심 사용 사례가 2인 면접이며, 여러 역할 편집 UI 없이 명확한 데이터 계약을 만들 수 있다.
**트레이드오프**: 패널 면접, 다대일 면접과 화자 분리 오류의 수동 교정은 지원하지 않는다.
**결과**: 화자가 두 명이 아니면 타임라인은 표시하지만 의미 검색은 비활성화한다.

## ADR-004: 연속 지원자 발화를 답변 구간으로 사용

**상태**: 승인
**결정**: 시간순으로 정렬한 발화에서 같은 지원자의 연속 발화를 하나로 묶고 다른 화자가 나오면 구간을 종료한다. 면접관의 짧은 맞장구도 우선 경계로 본다.
**이유**: RTZR 문단은 최대 130자의 읽기 단위이고 의미 검색의 답변 단위와 같지 않다. 확인되지 않은 맞장구 휴리스틱보다 단순하고 검증 가능한 규칙이 안전하다.
**트레이드오프**: 짧은 맞장구 때문에 하나의 긴 답변이 여러 구간으로 나뉠 수 있다.
**결과**: 실제 fixture에서 과도한 분할이 확인되기 전에는 맞장구 사전이나 시간 간격 병합 규칙을 추가하지 않는다.

## ADR-005: 10자 미만 답변을 검색에서만 제외

**상태**: 승인
**결정**: 공백과 문장부호를 제거한 텍스트가 10자 미만인 답변 구간은 타임라인에는 유지하지만 임베딩 검색에서는 제외한다.
**이유**: `네`, `맞습니다` 같은 짧은 응답의 노이즈를 줄이면서 다른 답변과의 자동 병합으로 의미가 왜곡되는 것을 피한다.
**트레이드오프**: 짧지만 중요한 답변이 검색에서 빠질 수 있다.
**결과**: 기준값은 한 곳에서 관리하고 실제 샘플 검증 결과 없이 임의 변경하지 않는다.

## ADR-006: EmbeddingGemma를 로컬 검색 모델로 사용

**상태**: 승인
**결정**: `google/embeddinggemma-300m`을 사용하고 질의에는 `Retrieval-query`, 답변에는 `Retrieval-document`, 양쪽 모두 `normalize_embeddings=True`를 적용한다.
**이유**: 정확한 문자열이 일치하지 않아도 평가 기준과 관련된 답변을 찾을 수 있고, 면접 전사문을 별도 생성형 AI 서비스에 전송하지 않아도 된다.
**트레이드오프**: 모델 사용 조건 동의, 토큰과 사전 다운로드가 필요하고 첫 준비 시간이 길 수 있다. 로컬 장치 성능의 영향을 받는다.
**결과**: MPS를 우선하고 실패 시 CPU로 한 번 대체한다. 답변 임베딩은 같은 전사·화자 선택에서 재사용한다.

## ADR-007: 모델을 앱 실행 전에 준비

**상태**: 승인
**결정**: `scripts/prepare_model.py`가 모델 다운로드, 실제 로드와 짧은 임베딩 생성을 검증한다. 앱은 `local_files_only=True`로 캐시된 모델만 로드한다.
**이유**: 사용자가 전사를 마친 뒤 모델 라이선스, 토큰 또는 다운로드 실패를 처음 발견하는 상황을 피한다.
**트레이드오프**: 설치 절차가 한 단계 늘고 앱이 런타임에 자동 복구하지 않는다.
**결과**: 모델이 없으면 준비 명령을 안내하며 앱에서 다운로드를 시작하지 않는다.

## ADR-008: 임계값 없는 Top 3와 순위 중심 표현

**상태**: 승인
**결정**: 코사인 유사도에 최소 임계값을 두지 않고 최대 3개를 내림차순으로 표시한다. 기본 UI는 순위를 보여주고 정확한 값은 상세 영역에 둔다.
**이유**: 모델과 데이터에 대해 검증된 절대 임계값이 없으며, 사용 목적은 자동 판정이 아니라 사람이 확인할 후보를 제공하는 것이다.
**트레이드오프**: 관련성이 낮은 결과가 포함될 수 있다.
**결과**: 화면에 유사도가 답변 품질이나 합격 가능성 점수가 아니라는 안내를 항상 표시한다.

## ADR-009: 답변 시작부터 종료 3초 후까지 공용 플레이어로 재생

**상태**: 승인
**결정**: 선택한 결과의 답변 시작 시점부터 마지막 발화 종료 3초 후까지 음원 경계 안에서 재생하며, Top 3 아래의 공용 플레이어 하나를 사용한다.
**이유**: 사용자가 선택한 답변을 즉시 재생할 수 있고, 전체 음원을 다시 탐색하는 부담과 반복 플레이어 UI를 줄인다.
**트레이드오프**: 더 긴 질문 문맥이 필요하면 타임라인이나 원본 음원에서 별도로 확인해야 한다.
**결과**: 자동 재생하지 않고 사용자가 재생을 시작한다.

## ADR-010: 결과를 메모리에서 JSON으로 직접 다운로드

**상태**: 승인
**결정**: 현재 질의와 Top 3의 제한된 필드만 UTF-8 JSON bytes로 만들어 `st.download_button`에 전달한다. 서버 파일을 만들지 않는다.
**이유**: 브라우저 다운로드와 서버 저장을 분리하고 면접 데이터의 불필요한 잔존을 막는다.
**트레이드오프**: 검색 기록, 전체 전사 백업과 서버 측 결과 목록을 제공하지 않는다.
**결과**: `outputs/`는 로컬 디버깅 전용이며 앱의 정상 사용자 흐름에 포함하지 않는다.

## ADR-011: 실제 API만 사용자에게 노출하고 fixture는 테스트에만 사용

**상태**: 승인
**결정**: 사용자 앱에는 저장된 전사 결과를 불러오는 샘플 모드를 만들지 않는다. 비식별 합성 완료 응답은 `tests/fixtures/rtzr_completed.json`에서 자동 테스트에만 사용한다.
**이유**: 제출 앱이 실제 RTZR API 활용을 명확히 보여주면서 반복 테스트는 API 사용량과 네트워크에 의존하지 않게 한다.
**트레이드오프**: 자격 증명이 없는 사용자는 앱의 검색 단계까지 체험할 수 없다.
**결과**: 실제 응답과 fixture는 같은 파서와 후처리 함수를 통과한다.

## ADR-012: 키워드 부스팅은 선택 기능으로 보류

**상태**: 승인
**결정**: RTZR 키워드 부스팅은 핵심 MVP를 완료한 뒤에만 선택 입력으로 고려하고 기본값은 비운다.
**이유**: 기술 용어의 전사 정확도 개선 가능성은 있지만 의미 검색과 별개이고, 입력을 바꾸면 재전사가 필요하다.
**트레이드오프**: MVP에서는 고유명사와 기술 용어의 오인식을 사용자가 보정할 수 없다.
**결과**: 구현 시 빈 줄·중복 제거, `sommers`용 한글 발음, 항목당 20자와 최대 500개 제한을 검증한다.

## ADR-013: Enterprise 기능과 자동 평가를 핵심 흐름에서 제외

**상태**: 승인
**결정**: RTZR 인사이트 같은 Enterprise 기능, 질문 자동 탐지, 여러 지원자 비교와 자동 점수화는 사용하지 않는다.
**이유**: 일반 개발자 계정에서 재현 가능해야 하며 제품의 역할은 판단이 아니라 근거 검색이다.
**트레이드오프**: 자동 요약·평가와 다중 지원자 워크플로는 제공하지 않는다.
**결과**: UI 문구, 데이터 모델과 export 스키마에 합격 판단 또는 평가 점수 필드를 추가하지 않는다.

## ADR-014: 오류 유형별로 재시도 정책 분리

**상태**: 승인
**결정**: 전사 생성 요청과 기존 작업 폴링을 서로 다른 멱등성 경계로 다룬다. 생성 요청의 네트워크 타임아웃은 자동 재시도하지 않고, `429 A0002`만 2·4·8초로 최대 세 번 재시도한다. 폴링의 일시적 오류는 동일 ID로 재시도한다.
**이유**: 생성 요청 타임아웃은 서버 접수 여부가 불명확해 중복 작업을 만들 수 있지만 조회는 동일 작업에 안전하게 반복할 수 있다.
**트레이드오프**: 일부 생성 실패는 사용자가 수동으로 재시도해야 한다.
**결과**: 60분 후에는 ID를 잃지 않고 자동 폴링만 멈추며 조회 재개를 제공한다.

## ADR-015: 사용자 흐름을 세 개의 depth로 분리

**상태**: 승인
**결정**: UX를 `음성 준비`, `화자 확인`, `근거 찾기`의 세 depth로 나눈다. 선택한 결과의 문맥과 오디오 재생은 네 번째 페이지가 아니라 `근거 찾기` 안의 상세 layer로 연다.
**이유**: 업로드·역할 지정·검색은 서로 다른 판단을 요구한다. 모든 기능을 한 화면에 노출하면 긴 전사문 때문에 핵심 행동이 밀리고 현재 단계가 불분명해진다.
**트레이드오프**: 이전 단계로 이동하는 상태 전이와 무효화 규칙을 구현해야 한다.
**결과**: 다음 depth에 필요한 조건이 충족되기 전에는 해당 화면을 렌더링하지 않는다. 이전 depth로 돌아갈 때 전사 결과는 유지하고, 지원자 화자 또는 새 음성이 바뀌면 종속된 검색 상태만 초기화한다.

## ADR-016: 흰색·회색·스카이블루 시각 체계와 판단 중립적 카피

**상태**: 승인
**결정**: 흰색과 회색을 기본 표면으로 사용하고 스카이블루를 주요 버튼, 현재 depth, 선택 상태와 링크의 메인 컬러로 사용한다. 화면 문구는 사용자 행동을 평범한 한국어로 설명하며 `AI 분석`, `인사이트 생성`, `최적의 답변`, `지원자 적합도`를 사용하지 않는다.
**이유**: 검토 도구의 차분한 성격을 유지하고, 임베딩 검색이 사용자를 자동 평가한다는 인상을 피하기 위해서다.
**트레이드오프**: 기술적 차별점을 마케팅 문구로 강하게 드러내지 않으며, 고정 문구와 색상 토큰을 일관되게 관리해야 한다.
**결과**: 주요 스카이블루는 `#0284C7`, hover는 `#0369A1`, 선택 배경은 `#E0F2FE`를 기준으로 한다. 결과는 `찾아볼 만한 답변`, 행동은 `관련 답변 찾기`, `이 구간 듣기`, `전사 문맥 보기`로 표현한다.
